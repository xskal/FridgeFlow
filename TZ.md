# Техническое задание на проект: «FridgeFlow»

## 1. Цель проекта

Создание веб-сервиса для планирования недельного меню и автоматической генерации списка покупок для семей и отдельных пользователей. Система решает три ключевые проблемы:

1. **Пищевые отходы** — по статистике, до 30% купленных продуктов выбрасывается из-за истечения срока годности или неправильного планирования.
2. **Временные затраты** — ежедневное принятие решения «что приготовить» съедает ментальный ресурс и время.
3. **Финансовые потери** — спонтанные покупки и отсутствие учета остатков приводят к перерасходу бюджета на питание.

Сервис автоматизирует процесс: пользователь один раз вносит остатки в «виртуальный холодильник», планирует блюда на неделю, а система сама рассчитывает, какие продукты нужно докупить, и формирует оптимизированный список покупок. Дополнительно предоставляется аналитика по расходам и пищевому балансу (КБЖУ).

## 2. Роли пользователей

### Гость
- Просмотр публичного каталога рецептов с фильтрацией по категориям, времени приготовления и сложности.
- Доступ к демо-версии аналитических графиков (без персональных данных).
- Регистрация в системе.

### Авторизованный пользователь
- Ведение личного «Виртуального холодильника» (учет остатков продуктов со сроками годности).
- Планирование меню на неделю через интерактивный календарь (завтрак/обед/ужин на каждый день).
- Автоматическая генерация списка покупок на основе плана питания и текущих остатков.
- Отслеживание покупок (отметка «куплено»).
- Просмотр персональной аналитики: расходы на питание по неделям, распределение КБЖУ, топ продуктов по стоимости.
- Получение внутренних уведомлений о продуктах, срок годности которых истекает в ближайшие 2 дня.

### Администратор (через Django Admin)
- Управление каталогом ингредиентов (добавление, редактирование, удаление).
- Модерация и добавление рецептов с указанием ингредиентов и инструкций.
- Просмотр статистики по пользователям и популярности рецептов.

## 3. Модели данных (сущности)

### Модель 1: `Ingredient` (Продукт)
Базовый справочник продуктов с пищевой ценностью и экономическими параметрами.
**Поля:**
- `name` — название продукта (CharField)
- `category` — категория: Овощи, Мясо, Молочка, Крупы и т.д. (CharField с choices)
- `unit` — единица измерения: кг, л, шт, уп (CharField)
- `price_per_unit` — цена за единицу (DecimalField)
- `calories_per_100g`, `protein_per_100g`, `fat_per_100g`, `carbs_per_100g` — пищевая ценность (DecimalField)
- `shelf_life_days` — типичный срок хранения в днях (IntegerField)

### Модель 2: `Recipe` (Рецепт)
Карточка блюда с инструкцией приготовления.
**Поля:**
- `name` — название рецепта (CharField)
- `description` — краткое описание (TextField)
- `cooking_time_minutes` — время приготовления в минутах (IntegerField)
- `servings` — количество порций (IntegerField)
- `difficulty` — сложность: Легко / Средне / Сложно (CharField с choices)
- `instructions` — пошаговая инструкция (TextField)
- `image` — изображение блюда (ImageField)

### Модель 3: `RecipeIngredient` (Ингредиент рецепта)
Промежуточная модель для связи M2M между рецептом и ингредиентами с указанием количества.
**Поля:**
- `recipe` — связь с Recipe (ForeignKey)
- `ingredient` — связь с Ingredient (ForeignKey)
- `quantity` — требуемое количество (DecimalField)

### Модель 4: `MealPlan` (План питания)
Запись о запланированном блюде на конкретный день и прием пищи.
**Поля:**
- `user` — владелец плана (ForeignKey на User)
- `date` — дата приема пищи (DateField)
- `meal_type` — тип: Завтрак / Обед / Ужин / Перекус (CharField с choices)
- `recipe` — выбранное блюдо (ForeignKey на Recipe)
- `servings_planned` — количество порций к приготовлению (IntegerField)

### Модель 5: `PantryItem` (Остаток в холодильнике)
Запись о конкретном продукте, имеющемся у пользователя, с контролем срока годности.
**Поля:**
- `user` — владелец (ForeignKey на User)
- `ingredient` — какой продукт (ForeignKey на Ingredient)
- `quantity` — текущее количество (DecimalField)
- `purchase_date` — дата покупки (DateField)
- `expiry_date` — срок годности (DateField)

### Модель 6: `ShoppingListItem` (Список покупок)
Элемент персонального списка покупок, сгенерированного автоматически или добавленного вручную.
**Поля:**
- `user` — владелец (ForeignKey на User)
- `ingredient` — что купить (ForeignKey на Ingredient)
- `quantity` — сколько купить (DecimalField)
- `is_purchased` — отмечено как купленное (BooleanField)
- `added_date` — дата добавления (DateTimeField, auto_now_add)

## 4. Ключевой функционал (User Stories)

### US-1. Каталог рецептов с фильтрацией
**Как** гость или пользователь, **я хочу** просматривать каталог рецептов с возможностью фильтрации по категории (завтрак/обед/ужин), времени приготовления и сложности, **чтобы** быстро находить подходящие блюда для планирования.

**Реализация:** ListView с фильтрами через django-filter, сортировка по популярности/времени, пагинация.

### US-2. Планирование меню на неделю
**Как** авторизованный пользователь, **я хочу** добавлять рецепты в интерактивный календарь на конкретные дни и приемы пищи (завтрак/обед/ужин), **чтобы** иметь готовый план питания на неделю и не думать ежедневно «что приготовить».

**Реализация:** Интерфейс на базе FullCalendar.js или кастомная сетка 7×3. Drag-and-drop или выбор через модальное окно. Валидация: нельзя добавить два обеда на один день.

### US-3. Учет остатков в «виртуальном холодильнике»
**Как** пользователь, **я хочу** вносить продукты, которые есть у меня дома, с указанием количества и срока годности, **чтобы** система учитывала их при генерации списка покупок и предупреждала о скором истечении срока.

**Реализация:** CRUD для PantryItem. Сортировка по сроку годности (красная подсветка для истекающих через 1-2 дня). Внутренние уведомления на дашборде.

### US-4. Автоматическая генерация списка покупок
**Как** пользователь, **я хочу** нажать одну кнопку и получить готовый список покупок на неделю, **чтобы** не считать вручную, сколько нужно курицы, если она используется в 3 разных рецептах.

**Реализация:** Алгоритм агрегации:
1. Суммируются все ингредиенты из MealPlan за выбранный период (с учетом количества порций).
2. Из полученного списка вычитаются остатки из PantryItem.
3. Результат записывается в ShoppingListItem с группировкой по категориям продуктов.

### US-5. Аналитика расходов и КБЖУ
**Как** пользователь, **я хочу** видеть графики своих расходов на питание за последние недели и распределение белков/жиров/углеводов, **чтобы** контролировать бюджет и следить за сбалансированностью рациона.

**Реализация:** Интерактивные графики Plotly:
- Столбчатая диаграмма «Расходы по неделям» (агрегация Sum по ShoppingListItem × Ingredient.price).
- Круговая диаграмма «Распределение КБЖУ за неделю».
- Топ-10 самых дорогих продуктов в рационе.

## 5. Внешние интеграции и аналитика

### Библиотеки анализа данных
- **Plotly** — построение интерактивных графиков (расходы, КБЖУ, топ продуктов). Графики встраиваются в шаблоны через `fig.to_html()`.
- **Pandas** — предварительная обработка и агрегация данных перед визуализацией (группировка по неделям, расчет средних, выявление аномалий в расходах).
- **Django ORM агрегации** — `Sum`, `Avg`, `Count`, `F` и `Q` объекты для сложных запросов (например, расчет общей стоимости плана питания с учетом связей).

### Внешнее API (опционально, для расширения функционала)
- **Spoonacular API** (https://spoonacular.com/food-api) — бесплатный тариф позволяет выполнять 150 запросов в день. Используется для:
  - Поиска рецептов по имеющимся ингредиентам («Что приготовить из курицы и риса?»).
  - Автоматического обогащения каталога рецептов (импорт с расчетом КБЖУ).
  - Генерации списка покупок из внешнего рецепта.

### Email-уведомления (бонусный функционал)
- Отправка писем пользователям о продуктах, срок годности которых истекает в ближайшие 2 дня (через Django management command + cron на PythonAnywhere).
